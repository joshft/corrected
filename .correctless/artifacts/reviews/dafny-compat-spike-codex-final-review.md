1. [severity: blocking] — INV-006, INV-009, INV-013, INV-014, DD-008, P01–P12 — Atomic report writes prevent partial files, but nothing proves that reports or phase receipts belong to the current run. If Route B crashes before replacing an old valid Route B report, the aggregator can consume the stale file and report both routes COMPATIBLE; P01 restore records and build-identity artifacts have the same replay problem. — Have the canonical runner create a unique run directory and unpredictable `run_id` before provisioning. Bind every restore/build/shared/route receipt to it, pass expected report paths directly from current child launches, and reject mismatched IDs or reports not produced by those launches. Exclude `run_id` from deterministic equality but require it for aggregation.

2. [severity: blocking] — INV-008, DD-001, INV-001, P01, P03 — The project topology is impossible as written. One package-owning `SpikeDafnyAdapter` project must reference the union of Route A and Route B dependencies, contaminating Route B with DafnyDriver’s graph and defeating route isolation. Conditional references would still collide with the one project lock. Moreover, `PrivateAssets="all"` prevents compile and runtime assets from flowing to the parent executable, so the route harness may lack its runtime dependencies. This `PrivateAssets` behavior is certain from Microsoft’s documentation; DafnyDriver’s distinct dependency graph is likewise confirmed by its [4.11.0 package metadata](https://www.nuget.org/packages/DafnyDriver/4.11.0). — Use one logical boundary but two implementation projects, such as `SpikeDafnyAdapter.RouteA` and `.RouteB`, sharing only Dafny-free contracts. Each implementation owns its route-specific packages and lock. Use the documented compile-only privacy pattern so runtime assets still flow, plus a negative compile test proving route consumers cannot name Dafny types. See [PackageReference asset controls](https://learn.microsoft.com/en-us/nuget/consume-packages/package-references-in-project-files).

3. [severity: blocking] — INV-013, INV-004, INV-005, INV-007, INV-011, INV-012 — The net8.0 adjudication rule is logically invalid: “fails on net8 too” means only “not net10-specific,” not “harness bug.” A missing typed observable, a Dafny 4.11 API limitation, or an upstream defect can reproduce identically on both runtimes. This also conflicts with INV-004’s rule that unavailable required observability becomes INCOMPLETE. — Split candidates into at least `HOST_RUNTIME_INCOMPATIBILITY`, `OFFICIAL_API_CAPABILITY_GAP`, and `HARNESS_FAULT`. Net8 success/failure may adjudicate only the first. Treat a demonstrably absent required official surface as a capability candidate, not an environment fault. Require a minimized official-surface reproducer and source/upstream control before either failure class becomes ADR-citable; otherwise retain `INCOMPATIBLE-candidate`.

4. [severity: blocking] — INV-013, EA-001, DD-003 — The net8.0 control has no controlled net8 runtime. `global.json` selects an SDK independently of the runtime, and a framework-dependent net8 application does not normally roll across the major-version boundary to net10. These claims are certain from Microsoft’s [global.json documentation](https://learn.microsoft.com/en-us/dotnet/core/tools/global-json) and [runtime roll-forward rules](https://learn.microsoft.com/en-us/dotnet/core/runtime-discovery/troubleshoot-app-launch). “Build the identical seam assembly for net8.0” is also impossible literally; changing TFM changes the assembly. — Specify identical source, package graph, options, and fixtures—not a bit-identical assembly—and provision or require a pinned net8 runtime control host. Record proof that it actually ran on 8.x. If that runtime is unavailable, adjudication remains INCOMPLETE rather than silently running the control on net10.

5. [severity: blocking] — INV-007, P11, INV-009 — The normalization-canary requirement assumes a post-pipeline option store independent of harness input. In Dafny 4.11, `Compilation.Options` is directly `Input.Options`, as shown in the tagged [Compilation source](https://github.com/dafny-lang/dafny/blob/v4.11.0/Source/DafnyCore/Pipeline/Compilation.cs). Thus simply reading it after verification is input echo by API design. No concrete option ID and verified upstream normalization are named, and an absolute solver-path canary conflicts with the report-wide repo-relative path rule. The alias claim is certain; existence of a suitable normalized field is unproven. — Name one concrete, source-verified transformation per route before TDD. Record requested value, the exact options object used, its state after upstream solver-option processing, and a behavioral differential proving the effective value. If no such typed post-processing observation exists, report a capability candidate instead of inventing a canary.

6. [severity: major] — INV-003, EA-005, P05, P08, P09 — The decoy test contradicts the startup gate: the test must plant `z3-4.12.1` under a route output tree, while the gate must abort if any such executable exists there. The decoy also has no stated observation channel, and “zero invocations” cannot be proven from a recording created only when the stub runs. Separately, official Dafny material describes `PROVER_PATH` as a Boogie prover option and `PATH` as the environment fallback; treating `PROVER_PATH` as an environment variable is a high-confidence inference error, not a confirmed behavior ([Dafny solver-path documentation](https://dafny.org/blog/2023/03/03/dafny-4-released/)). — Make nonce-recording assembly-adjacent and first-on-PATH decoys expected fixtures of every real run, and assert zero decoy invocations. Pre-create a nonce-bound ledger with count zero before P08/P09. Inspect and sanitize the typed Boogie prover-option list for duplicate `PROVER_PATH` values; drop the environment-variable rule unless source inspection confirms it.

7. [severity: major] — INV-006, INV-013, INV-014, Probe Manifest, P02, P04 — Shared-probe ownership conflicts with both the 22-entry total and the child exit-code contract. If each route reports P02/P04, the run has duplicate shared keys; if only the aggregator owns them, a route child cannot truthfully exit as `COMPATIBLE(route)`. A real crash also cannot guarantee a prescribed “crashed-before-report” exit code, and missing/malformed reports have no “first failing probe.” — Let route children report only route-probe status. The aggregator owns shared probes and alone computes COMPATIBLE. Map arbitrary signals/unknown exit codes plus absent or malformed reports to `crashed-before-report`. Replace mandatory `first failing probe` with a typed reason union supporting probe failure, prerequisite failure, missing report, malformed report, and crash. Reject duplicate keys in the raw JSON array before dictionary conversion.

8. [severity: major] — INV-002, INV-009, P03, DD-001 — “Loaded-assembly set equality” has no defined universe. Equality over every loaded assembly conflicts with floating runtime patches and lazy framework loads; equality over a hand-picked package subset can omit a route’s defining assembly. File hash plus informational version also does not prove that the loaded file came from the locked, isolated NuGet asset rather than a locally substituted binary. — Define the universe as trust-relevant managed runtime assets selected through the route’s `.deps.json`, excluding shared-framework assemblies. Require route anchors—at minimum DafnyDriver/Core for A and a meaningful DafnyPipeline/Core touch for B—then trace every loaded package assembly through `.deps.json` to the isolated package asset and compare hashes. Any loaded Dafny/Boogie-family assembly outside that mapping must fail P03.

9. [severity: major] — INV-001, INV-002, PRH-002, BND-001 — Several NuGet/MSBuild claims are technically wrong or contradictory. Plain `Version="4.11.0"` is a minimum-inclusive NuGet range, not an exact constraint; use `[4.11.0]` if declarative exactness is intended. An “empty `Directory.Packages.props`” cannot also contain the central System.CommandLine transitive pin. Empty `Directory.Build.props` does not block independently discovered `Directory.Build.targets` or `Directory.Build.rsp`. These are certain from [NuGet PackageReference rules](https://learn.microsoft.com/en-us/nuget/consume-packages/package-references-in-project-files), [central transitive pinning](https://learn.microsoft.com/en-us/nuget/consume-packages/central-package-management), and [MSBuild directory customization](https://learn.microsoft.com/en-us/visualstudio/msbuild/customize-by-directory). The stale-lock negative test also accepts any restore failure, including network or missing-package failure, as proof of locked mode. — Require exact bracket constraints, a non-empty authoritative spike-local package file, sealing files for props/targets/response arguments, and NuGet downgrade warnings as errors. Run a successful control restore first, then mutate only a lock consistency field and require the specific locked-mode mismatch diagnostic.

10. [severity: major] — INV-007, P10, PRH-004 — The sidecar-shape test still self-certifies resolver-inferred ghostness. A sidecar saying `inferred-ghost=true` and `explicit-keyword=false` does not prove the fixture lacks the keyword or that the resolver inferred the value. — Add an independent AST test that obtains explicit-modifier presence from the pinned parser AST and ghostness from the resolved AST, without consulting the sidecar. A useful differential is to mutate a copy by adding `ghost`: resolved ghostness should remain true while `inferred-ghost` changes to false.

11. [severity: major] — INV-009, INV-010, DD-006, DD-008 — The evidence schema controls the deterministic projection but is bound only by a version string. Editing the schema can reinterpret an old report or silently omit a newly added field unless every nested schema is closed and classification is total. This leaves the three-way partition weaker than the hard-pinned probe manifest. — Include the schema SHA-256 in every report, keep immutable versioned schemas, set `additionalProperties: false` recursively, and make projection construction fail if any instance field lacks exactly one classification. Bind the Option Manifest and other oracle schemas by digest as well.

12. [severity: major] — INV-006, DD-009, STRIDE/TB-004 — The stated 30-minute “full run” bound is not actually enforced as a total bound. Independent route timeouts can consume 60 minutes, and provisioning, restore, build, shared probes, aggregation, or the test process itself can hang outside either child timeout. — Give the canonical runner one monotonic deadline covering the explicitly defined run phases. Pass remaining time to each child and place the runner itself under one outer watchdog. State whether online provisioning/restore count; “full run” currently says they do.

13. [severity: major] — PRH-004, INV-014, EA-007, EA-008 — The one-launcher rule has a bootstrap cycle. The shell entry point must invoke `dotnet`, downloading, hashing, extraction, and build commands before the managed launcher exists, yet `dotnet` and concrete provisioning utilities are not enumerated. “Provisioning fetch/verify tooling” is a category, not an allowlist. — Define two explicit layers: a minimal bootstrap script allowlist with exact executable names/arguments, and the managed launcher for post-build test processes. Sanitize the bootstrap environment through an allowlist rather than the untestable “no MSBuild env overrides” prohibition.

14. [severity: major] — Complexity Budget, DD-001, DD-009, INV-003, INV-006, INV-013 — The 1,600–2,400 C# estimate is no longer credible. Two route-specific adapter implementations, a genuine net8 control, run-coherent orchestration, sentinel/decoy executables, schema projection, atomic reports, process supervision, and the induced-failure matrix are more plausibly 3,000–5,000 C# LOC plus scripts and fixtures. That makes ten working days an evidence-discovery time-box, not a likely complete implementation window. — Re-budget before TDD. Do not cut the manifest equality/deletion tests, route isolation, non-vacuity, solver identity, aggregator, or induced-failure subset—the spec explicitly makes that armor non-trimmable. Simplify by using one runner and unique run directory, one always-on decoy mechanism, local canned provisioning-state tests plus one live E2E, and by dropping nonessential exact-line/hostfxr/CI-stub polish from the compatibility verdict.

15. [severity: minor] — INV-005, INV-014, DD-005, DD-007 — Several requirements are stricter or less structural than claimed. Exact diagnostic line equality is not required by Phase 0.0 bullets 1–3 and can reject a working route over diagnostic granularity; a CI stub’s existence does not force later CI wiring; and assigning ADR promotion to an unnamed future “final feature” is prose, not enforcement. — Accept an upstream diagnostic range containing the planted assertion unless exact-line identity is a separately justified product requirement. Give deferred obligations stable machine-checked IDs that future feature completion must discharge, rather than treating placeholder files as gates.

Overall verdict: the spec is not ready for TDD. Its honesty mechanisms are substantially stronger than v2’s, but four blocking defects remain: stale-report replay, the impossible single-seam/two-route topology, invalid net8 adjudication, and an unsupported options-readback canary. After those are corrected, the top residual risks are the still-unproven Dafny typed observation surfaces, trustworthy solver execution identity without excessive harness machinery, and delivery inside the ten-day time-box.
