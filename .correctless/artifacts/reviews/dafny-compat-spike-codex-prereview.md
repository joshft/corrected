# Disposition: all 16 findings accepted (some adapted for spike proportionality); spec revised in same session

I’ll first load the project’s mandated Correctless context, then verify the toolchain-specific claims against Dafny/NuGet/Z3 primary sources. I’ll focus especially on ways the probe could pass while exercising the wrong assemblies, solver, source closure, or API path.
The project context confirms the narrow scope, but it also sharpens the acceptance bar: “package present” is not enough. I’m now checking whether the probes bind to the assemblies and solver process actually loaded, and whether the chosen driver route can quietly bypass the package surface the ADR would claim compatible.
1. [severity: blocking] — INV-006/PRH-001/DD-001 — The required probe plan is self-declared and shown with an ellipsis. Removing a probe from both the declared and completed sets still produces equality and permits COMPATIBLE; even an empty plan can satisfy the stated rule. — Define a versioned, spec-owned manifest enumerating every mandatory probe ID, route-specific probe, and expected outcome. Require non-empty exact set equality, reject duplicates/unknown IDs, include the manifest digest in evidence, and test deletion of each required ID.

2. [severity: blocking] — INV-003/INV-004/INV-005/INV-009/PRH-004/BND-002 — Pre-running and hashing a Z3 file does not prove Boogie executed that file. A misapplied `--solver-path` could fall back to another Z3 while the report records the prechecked binary; “nonzero solver duration” is not execution identity. — On linux-x64, capture the actual child-process execution path and digest using an instrumented launcher or process-exec trace. Run with ambient solver discovery disabled, add a sentinel solver-path test proving the option is honored, and require the child-exec allowlist to contain only the instrumented launcher and pinned Z3—never Dafny CLI. The Dafny 4.11 manual confirms that `--solver-path` selects the external SMT solver. ([Dafny 4.11 reference](https://dafny.org/v4.11.0/DafnyRef/DafnyRef)) Toolchain basis: certain.

3. [severity: blocking] — INV-004/INV-006/BND-003 — “Every proof obligation verified” is vacuously true when zero obligations are produced. Parse errors normalized as an empty program, an omitted file, a bodyless declaration, or an accidental verification filter can therefore pass the positive probe. — Fix the expected parsed declarations, named `ICanVerify` targets, positive VC/query count, and Boogie result count; require zero parse/resolution errors and an observed Z3 launch. Structurally assert that `ok.dfy` has bodies and contains no attributes, assumptions, or suppression controls.

4. [severity: blocking] — INV-007/BND-003 — The authoritative bullet requires recovery of proof versus executable nodes, but the fixture only requires declaration kinds and variables. It omits the Phase 0.1 node forms—`assert`, `calc`, loop invariants, and `decreases`. Its explicitly ghost declarations also do not demonstrate resolver inference. — Add an exact resolved-node table covering all four editable-proof forms plus executable assignments/control flow and an `expect` contrast. Add at least one variable without a `ghost` keyword whose RHS makes it resolver-inferred ghost. Assert resolved AST types/properties and source spans, never syntax text. Dafny 4.11 explicitly performs ghost inference after type inference and marks ghost-RHS variable declarations, `assert`, and `calc` as ghost while `expect` remains non-ghost. ([Dafny 4.11 ghost inference](https://dafny.org/v4.11.0/DafnyRef/DafnyRef)) Toolchain basis: certain.

5. [severity: major] — INV-001/INV-002/INV-009/BND-001 — Lock-file versions are being reported as identities of what ran. That does not prove which assemblies the console process loaded, and the asserted pins do not constrain every Dafny/Boogie family member. DafnyCore and Boogie packages publish lower-bound dependencies (`>=`), allowing a coherent-looking lock with mixed newer components. ([DafnyCore dependencies](https://www.nuget.org/packages/DafnyCore), [Boogie dependencies](https://www.nuget.org/packages/Boogie.ExecutionEngine)) — Require all resolved `Dafny*` and `Boogie.*` packages to match an explicit family-version policy. At runtime, record and compare each loaded assembly’s name, informational version, MVID, SHA-256, and normalized location against the lock, `.deps.json`, and package content hash. Toolchain basis: certain.

6. [severity: major] — DD-001/INV-001/INV-004/INV-006 — Driver-first fallback cannot be evaluated cleanly in one dependency graph. If the project references DafnyDriver, its LanguageServer/TestPlatform closure remains present during the supposed Core/Pipeline fallback and may fail before fallback logic executes. Conversely, DafnyDriver does not depend on DafnyPipeline, so a successful Driver route may never load the package the verdict names. ([DafnyDriver dependencies](https://www.nuget.org/packages/DafnyDriver/), [DafnyPipeline dependencies](https://www.nuget.org/packages/DafnyPipeline/)) — Use separate minimal executables/projects, locks, `.deps.json` files, and probe plans for Core/Pipeline and Driver routes. Record both outcomes and issue only a boundary-qualified verdict for the independently passing route. Toolchain consequence: inferred from the certain package graphs.

7. [severity: major] — INV-004/INV-005/INV-007/DD-003 — “Equivalent test-host invocation” is not equivalent to running the console harness. A test host supplies a different runtime configuration and dependency graph and can either hide or introduce assembly conflicts. — Require integration tests to launch the built console artifact in a clean process using its own `.runtimeconfig.json` and `.deps.json`, after locked restore/build with `--no-restore`. The test process should only validate the emitted report. Toolchain basis: certain for .NET dependency-host behavior.

8. [severity: major] — INV-004/INV-005/INV-006/INV-007 — Only successful parse/resolution paths are probed. The ADR nevertheless promises failure behavior, and an adapter that conflates parse failure, resolution failure, and empty verification is untested. — Add malformed-syntax and unresolved-symbol fixtures. Require typed, stage-specific diagnostics with exact source locations, zero verifier targets, no Z3 launch, and non-COMPATIBLE outcomes. Also require `bad.dfy` to fail specifically from a completed solver result, not infrastructure or frontend failure.

9. [severity: major] — INV-007/INV-009/DD-001 — “DafnyOptions relevant to verification” and “effective-options set present” have no defined membership or expected values. An implementation can serialize its input object, omit dangerous options, or report values different from those used by verification. Dafny exposes scope-changing options such as `--filter-symbol`, `--verify-included-files`, `--library`, `--no-verify`, time limits, and `--solver-path`. ([Dafny 4.11 options](https://dafny.org/v4.11.0/DafnyRef/DafnyRef)) — Freeze an exact option-ID/value/provenance manifest and assert set equality. Include verification enablement, filters, included/library behavior, solver path, cores, time/resource controls, function syntax, project/config inputs, and Boogie prover options. Prove the same effective options instance or canonical projection reaches parse, resolution, and verification. Toolchain basis: certain.

10. [severity: major] — INV-007/BND-003 — A source set expected to contain only the supplied file cannot distinguish real closure recovery from simply echoing the input argument. It also conflates parsed closure with verified files. Dafny parses the transitive include closure but does not verify included files by default. ([Dafny 4.11 include semantics](https://dafny.org/v4.11.0/DafnyRef/DafnyRef)) — Add a spike-only two-file include fixture, even though includes remain unsupported in Phase 0.1. Assert separately the root inputs, parsed/resolved closure, and verification-target set under both values of `--verify-included-files`. Toolchain basis: certain.

11. [severity: major] — EA-001/INV-001/INV-009/BND-001 — EA-001’s `10.0.1xx` assumption is stale and no exact SDK selection is specified. As of the specification date, the supported release is SDK 10.0.302/runtime 10.0.10; Microsoft specifically recommends `global.json` with roll-forward disabled when package locks are used. ([.NET 10 downloads](https://dotnet.microsoft.com/en-us/download/dotnet/10.0), [`global.json` guidance](https://learn.microsoft.com/en-us/dotnet/core/tools/global-json)) — Pin one exact supported SDK with `allowPrerelease: false` and `rollForward: disable`; record SDK, host/fxr, actual CoreCLR, architecture, RID, and non-identifying libc/platform version. Toolchain basis: certain.

12. [severity: major] — INV-009/INV-010/DD-006 — The committed evidence is not bound to the harness implementation or required to match a fresh run. A stale or manually edited sample can support ADR claims while current probes pass independently. — Add hashes for the harness binary/source tree, probe manifest, package lock, `.deps.json`, `.runtimeconfig.json`, fixtures, selected route, and evidence schema. Require the committed deterministic projection to equal a fresh run before ADR acceptance.

13. [severity: major] — INV-001/BND-001 — “Only nuget.org” is asserted but not enforced. NuGet normally composes machine, user, and repository configuration and can use fallback/global package folders; locked mode alone does not prove the initial source. ([NuGet configuration precedence](https://learn.microsoft.com/en-us/nuget/consume-packages/configuring-nuget-behavior), [package source mapping](https://learn.microsoft.com/en-us/nuget/consume-packages/package-source-mapping)) — Commit a spike-local `NuGet.Config` with `<clear/>`, only nuget.org, and source mapping; invoke restore with `--configfile`; use a spike-local global-packages folder for the evidentiary restore; record effective sources and content hashes. Toolchain basis: certain.

14. [severity: major] — DD-004/DD-005/DD-006/INV-009 — ADR-0001 risks presenting a bullets-1–3 result as the final Phase 0.0 integration decision even though bullets 4–12 can still invalidate the structural boundary. — Make the ADR status provisional and name the verdict `COMPATIBLE_FOR_PHASE_0_0_BULLETS_1_3` or equivalent. List every deferred gate explicitly; only promote the ADR to accepted after the remaining Phase 0.0 features pass.

15. [severity: minor] — INV-008/PRH-004/INV-003 — Grep does not enforce either the Dafny seam or the subprocess prohibition: fully qualified names, aliases, reflection, or helper wrappers bypass it. It is also self-contradictory because INV-003 requires the harness to execute `z3 --version`, normally using the same process API being flagged. — Put the seam in a separate assembly that alone references Dafny/Boogie packages, enforce forbidden assembly/symbol references with a compiler analyzer or architecture test, and replace `Process.Start` grep with a single allowlisted process-launch component plus the runtime exec audit from finding 2.

16. [severity: minor] — INV-002 — “Dafny’s pinned beta” is factually inaccurate for consumers: the NuGet dependency is `System.CommandLine >= 2.0.0-beta4.22272.1`, not an exact package constraint. ([DafnyCore dependency metadata](https://www.nuget.org/packages/DafnyCore)) — Reword it as the required resolved version and enforce it through the full graph/loaded-assembly checks. If an exact dependency constraint is desired, use an explicit central transitive pin or exact PackageReference rather than simultaneously prohibiting every direct pin. Toolchain basis: certain.

Overall verdict: not ready to authorize a COMPATIBLE verdict. The spike’s intent and narrow scope are sound, but the current specification can pass with a self-shrunk probe plan, zero verification obligations, the wrong solver or child process, and without exercising the required proof/executable structural surface. Closing the four blocking findings and binding the verdict to an isolated route, actual loaded assemblies, actual Z3 execution, and a provisional ADR would make discovered incompatibility—and compatibility—credible.
