1. [severity: blocking] — INV-009, INV-010, DD-008, P01, P05, P11; F1/F11 interaction — The run-isolation fix makes deterministic equality impossible. Packages, Z3, and other products live under `out/<run_id>/`, while class 2 includes concrete restore argv, solver path, and Option Manifest readback. Those values necessarily contain a different `run_id` on each run, so INV-010 cannot pass. — Store concrete paths/argv in the binding class; compare class-2 paths after canonicalizing the run root to a fixed token such as `<run-root>`. Let the Option Manifest contain that token, expanding it for execution and normalizing it for evidence. Add a test proving no `run_id` occurs anywhere in the deterministic projection.

2. [severity: blocking] — INV-013, EA-001, EA-008, BND-001/BND-002, DD-009; F4 — The net8 control introduces a new runtime supply-chain input without specifying its exact patch, RID asset, source URL, digest, isolated installation, or host-resolution mechanism. “Proof it ran on 8.x” is weaker than “the pinned runtime ran.” This is certain: `global.json` selects an SDK, not an application runtime, and framework-dependent applications ordinarily roll to the latest installed compatible patch. [Microsoft’s SDK documentation](https://learn.microsoft.com/en-us/dotnet/core/tools/global-json), [runtime servicing rules](https://learn.microsoft.com/en-us/dotnet/core/releases-and-support). The asserted equality of source, graph, options, and fixtures is also not mechanically enforced; TFM conditionals could make the net8 executable materially different. A substituted or different-patch control could misadjudicate a failure. — Add a .NET-runtime intake boundary: exact runtime patch/RID/URL/SHA-256, isolated `DOTNET_ROOT`, exact host or `--fx-version`, and runtime-module identity evidence. Require normalized source, resolved package graph, option, fixture, and control-asset digest equality; forbid TFM-conditional harness logic. Make control-host unavailability block only a host-runtime candidate, not an independently demonstrated API capability gap.

3. [severity: blocking] — INV-004, INV-005, INV-007, INV-011, INV-012, INV-013, DD-007; F3 — The three-way taxonomy is neither exhaustive nor correctly typed. `HARNESS_FAULT` is placed under “INCOMPATIBLE-candidate” even though it must remain INCOMPLETE; `HOST_RUNTIME_INCOMPATIBILITY` asserts net10 specificity before the control establishes it; and there is no category for an official surface that exists but has an upstream implementation defect on both runtimes. Requiring a “typed exception” also cannot represent a wrong typed outcome or a missing observable, neither of which necessarily throws. Finally, the candidate-to-citable transition is only an advisory ADR review. — Begin with `UNADJUDICATED_IN_PROCESS_FAILURE`; use variant payloads for exception, unexpected typed result, and absent capability. Define terminal transitions mechanically: harness fault → INCOMPLETE; proven absent required surface → INCOMPATIBLE; net10 failure plus equivalent net8 success and minimized reproducer → host-runtime INCOMPATIBLE; reproducible cross-runtime upstream defect → a separate upstream-integration category. Require a schema-valid adjudication record before ADR rejection language is permitted.

4. [severity: major] — Probe Manifest P01/P02, INV-001, INV-006, INV-013; F7 — Probe ownership is still false to execution topology. P01 is labeled a route probe and therefore reported by the child, but restore occurs before that child exists. A child cannot authoritatively attest the runner’s locked restore. P02 is one shared entry but does not define how it checks build and actual runtime identity for both route executables, rather than only the aggregator. — Make the runner own P01 once per route and P02/P04 as shared probes; children should own P03 and P05–P12. P01 must bind every project lock in that route graph, exact argv, exit result, and generated assets to `run_id`. P02 must aggregate build records and runtime observations from every launched executable.

5. [severity: major] — INV-006, INV-009, INV-013; F7 — Verdict-reason and exit handling remain contradictory. INV-009 still requires “first failing probe + stage,” although INV-013 correctly says missing/malformed reports have none. INV-013 simultaneously provides `missing-report` and `malformed-report` variants but says both map to `crashed-before-report`. No rule rejects a known probe-failure exit accompanied by an all-pass report, which could yield COMPATIBLE. “First” is also nondeterministic if probes run concurrently. — Define one discriminated reason union with variant-specific required fields; choose probe failures in manifest order, never completion order. Add a complete exit/report consistency table. Any exit/report mismatch, unknown exit, signal, absent report, or malformed report must be non-COMPATIBLE with its precise reason.

6. [severity: major] — INV-001, INV-002, PRH-002, BND-001; F9 — The central package layout is technically inconsistent. Once authoritative CPM is enabled, versions belong on `PackageVersion` entries in `Directory.Packages.props`; project `PackageReference` entries omit `Version`. The spec instead mandates bracket versions on `PackageReference`. [Microsoft’s CPM documentation](https://learn.microsoft.com/en-us/nuget/consume-packages/central-package-management) is explicit about this. PRH-002 also bans “range” syntax while INV-001 mandates singleton range syntax `[x]`, and its csproj-only scan would miss the authoritative props file. — Require `ManagePackageVersionsCentrally=true`, `CentralPackageTransitivePinningEnabled=true`, and preferably disable `VersionOverride`; put exact `[x]` constraints on every applicable `PackageVersion`, with versionless `PackageReference` items. Define PRH-002 as forbidding non-singleton ranges and scan both project and central package files.

7. [severity: major] — INV-006, PRH-004, EA-007, EA-008, DD-008; F12/F13 — The two-layer launcher is still not an executable allowlist. “The download tool,” “sha256 utility,” and “archive extractor” are categories, not exact names; the shell and required filesystem/process-supervision utilities are omitted. The environment allowlist does not name `PATH`, `DOTNET_CLI_HOME`, runtime-root, locale, temporary-directory, or certificate variables, although the PATH decoy requires a constructed PATH. The component holding the pre-provisioning monotonic deadline is also unclear because the managed runner is not built yet. Finally, `dotnet restore/build/test` can spawn MSBuild nodes, compiler servers, and test hosts despite Boogie being described as the only indirect-spawn exception. — Select concrete bootstrap executables and resolution rules; define the policy as either direct launches only or include documented SDK-managed descendants. Name the bootstrap script as the pre-build deadline owner and specify its monotonic mechanism and synthetic-INCOMPLETE behavior. Commit the complete environment-key/value-normalization manifest and disable persistent build servers where practical.

8. [severity: major] — INV-008; F2 — The seam API test conflicts with the Dafny-free test-host claim and is incomplete. Ordinary reflection loads the seam and may resolve Dafny dependencies, while the test project supposedly never loads them. The enumerated surface omits properties, events, interfaces, delegates, generic arguments/constraints, nested public types, and recursively exposed types; a Dafny type can therefore leak while the test passes. The compile-only asset-flow approach itself is sound—NuGet documents privacy of compile assets while allowing runtime assets to flow. [NuGet asset controls](https://learn.microsoft.com/en-us/nuget/create-packages/select-assemblies-referenced-by-projects). — Use metadata-only inspection from the Dafny-free test process, or a separately launched inspector. Recursively inspect every externally visible metadata signature and add negative compile cases for properties, generic constraints, and interfaces.

9. [severity: major] — INV-009, INV-010, DD-006, DD-008; F11 — A report-provided schema digest is not yet anchored to the schema the aggregator is required to trust. The aggregator is not explicitly required to compute the committed schema’s digest before parsing and reject any other digest. Schema/oracle digests are also absent from the prose’s three field-class lists, so reports under different schemas could potentially compare equal. “Immutable versioned” remains unenforced. — Anchor the schema digest in the manifest/test constant or a separate append-only schema registry. Validate that digest before report projection or interpretation. Put schema and oracle digests in class 2, forbid reuse of a version with a new digest, and add schema-substitution and same-version-mutation tests.

10. [severity: major] — P11, INV-007, INV-013; F5 — The canary remains an unresolved design variable deferred to “RED design,” despite the spec claiming no open questions. The relevant source fact is certain: `Compilation.Options` directly returns `Input.Options`. [Dafny 4.11.0 source](https://github.com/dafny-lang/dafny/blob/v4.11.0/Source/DafnyCore/Pipeline/Compilation.cs). Without a named option, requested/effective values, source location, and behavioral oracle, “source-verified transformation” is not a test specification and can still be satisfied by an ad hoc observation. — Before approval, name each route’s exact canary and expected differential. If that research is intentionally part of RED, declare it as a bounded open adjudication task whose only allowed outcomes are the named canary contract or OFFICIAL_API_CAPABILITY_GAP.

11. [severity: major] — P03, INV-002, DD-001; F8 — Route B’s “meaningful DafnyPipeline load” is undefined. The package is a net8 library depending only on DafnyCore, and its tagged project principally packages standard-library/runtime resources. [NuGet package metadata](https://www.nuget.org/packages/DafnyPipeline/), [tagged project source](https://github.com/dafny-lang/dafny/blob/v4.11.0/Source/DafnyPipeline/DafnyPipeline.csproj). A bare `Assembly.Load` can therefore satisfy the current anchor while proving no required Pipeline behavior. — Name the exact Pipeline API/resource consumed on the verification path and add a removal/differential test proving that consumption matters. Otherwise state that Route B proves DafnyCore only and cannot support an ADR claim that DafnyPipeline was exercised.
